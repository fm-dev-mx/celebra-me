# RSVP Module Architecture (Multi-tenant)

**Last Updated:** 2026-03-17

## Context

The RSVP module supports:

- host-side guest management from the authenticated dashboard,
- guest-side personalized invitation access and RSVP submission,
- realtime dashboard refresh through SSE,
- strong session and MFA protections for sensitive operations.
- automated ViewModel adaptation for multi-tenant invitations.

The active stack is:

- Astro routes and API endpoints
- React islands for dashboard and RSVP interactions
- Supabase-backed persistence and authorization services

## Goals

- Hosts can CRUD guests under events they manage.
- Guests can open a personalized invitation experience from a direct or short URL.
- Guests can confirm or decline within server-enforced attendee limits.
- Dashboard updates remain near-real-time.
- Legacy RSVP data can coexist while the guest invitation model remains the active flow.

## Boundaries

- Client code never uses service role credentials directly.
- Public guest operations run through server APIs and accept `inviteId`.
- Host dashboard operations run through authenticated APIs under `/api/dashboard/**`.
- Legacy `/admin/rsvp` remains operational as a separate historical/operational surface and not the
  primary host dashboard flow.

## Active URL Patterns

### Host-Facing

- Dashboard entrypoint: `/dashboard/invitados`
- Guest dashboard APIs: `/api/dashboard/guests/**`
- Admin APIs for cross-entity operations: `/api/dashboard/admin/**`

### Guest-Facing

- Direct invitation URL:
  `/{eventType}/{slug}/invitado?invite={inviteId}`
- Short invitation URL:
  `/{eventType}/{slug}/i/{shortId}`

Current short-link behavior:

1. `/{eventType}/{slug}/i/{shortId}` resolves the short code server-side.
2. The route redirects to `/{eventType}/{slug}?invite={inviteId}` for canonical invite resolution.
3. Personalized guest UI is also available at `/{eventType}/{slug}/invitado?invite={inviteId}` and
   is the direct URL generated by `generateInvitationLink()`.

Because both patterns exist in the live tree, documentation must treat the guest experience as a
dual-entry system rather than a single `/invitation/{inviteId}` route.

## Data Model

Primary v2 tables:

- `host_profiles`
- `events`
- `guest_invitations`
- `guest_invitation_audit`
- `app_user_roles`
- `event_memberships`
- `event_claim_codes`

Legacy tables retained during transition:

- `rsvp_records`
- `rsvp_audit_log`
- `rsvp_channel_log`

## Data Flow

### Host Flow

1. Host signs in through Supabase-backed auth flows.
2. Unauthenticated access to `/dashboard/invitados` redirects to `/login?next=/dashboard/invitados`.
3. Dashboard UI calls `/api/dashboard/guests` and related dashboard endpoints.
4. Services resolve guest/event operations through `src/lib/rsvp/**`.
5. CRUD actions update `guest_invitations` and append `guest_invitation_audit`.
6. Dashboard listens to `/api/dashboard/guests/stream` for near-real-time refresh.

### Guest Flow

1. Guest opens either a direct personalized URL or a short URL.
2. Short URLs resolve server-side and redirect to the invitation experience.
3. Guest context is loaded from `/api/invitacion/:inviteId/context`.
4. View telemetry posts to `/api/invitacion/:inviteId/view`.
5. RSVP submissions post to `/api/invitacion/:inviteId/rsvp`.
6. Server-side validation enforces attendee bounds, sanitization, and status rules.

## API Contracts

### Host API

- `GET /api/dashboard/guests?eventId=...&status=...&search=...`
- `POST /api/dashboard/guests`
- `PATCH /api/dashboard/guests/:guestId`
- `DELETE /api/dashboard/guests/:guestId`
- `POST /api/dashboard/guests/:guestId/mark-shared`
- `GET /api/dashboard/guests/export.csv?eventId=...`
- `GET /api/dashboard/guests/stream`

### Auth API

- `POST /api/auth/login-host`
- `POST /api/auth/register-host`
- `GET /api/auth/session`
- `POST /api/auth/logout`
- `POST /api/auth/sync-session`
- `POST /api/auth/refresh-session`

Registration policy:

- Host signup requires `claimCode` plus email.
- Successful claim creates event membership for the new host account.
- `eventSlug` may be accepted as compatibility input, but claim resolution is driven by the claim
  code record.

### Guest API

- `GET /api/invitacion/:inviteId/context`
- `POST /api/invitacion/:inviteId/rsvp`
- `POST /api/invitacion/:inviteId/view`
- `GET /api/invitacion/resolve`

Historical note:

- `/api/invitation/:inviteId/*` is not the current public API contract.

## Realtime Strategy

Current implementation uses **Server-Sent Events (SSE)** via `/api/dashboard/guests/stream` to
provide near-real-time dashboard updates for guest status changes and audit activity. The module is
structured so a future websocket-based channel could be introduced without redefining domain
boundaries.

## Security

- RLS and authorization boundaries enforce event ownership and administrative access constraints.
- MFA is required for superadmin and sensitive dashboard operations.
- Session persistence uses server-side cookie synchronization to preserve elevated AAL2 state.
- Public guest endpoints never expose cross-event listings.
- Input sanitization and attendee bound checks are enforced server-side.
- Audit trail tracks invite lifecycle events such as view, response, and share.

> **Security Note (2026-03-17):** The current implementation still uses `useServiceRole: true`
> across multiple repository operations. This remains a known deviation from the ideal "host
> operations use authenticated user JWT and RLS" target and should stay documented until the service
> layer is fully refactored.

## Migration Notes

- `src/content/events/*.json` and `src/content/event-demos/**.json` remain active content inputs for
  invitation rendering.
- Dashboard source of truth is `guest_invitations`.
- Legacy operational surfaces such as `/admin/rsvp` remain available but are not the primary

## Design Patterns

- **Separation of Concerns**: RSVP logic is decoupled from UI presentation. Logic resides in `RSVPLogic.ts` (shared state and validation) while components in `RSVPComponents.tsx` and `RSVPFormFields.tsx` handle rendering.
- **Type-Safe Domain Adapters**: Event data is transformed via adapters (`src/lib/adapters/event.ts`) into a common `InvitationViewModel` to ensure consistent rendering across different event types.
- **Hardenened Middleware**: Security middleware (`src/middleware.ts`) enforces guest access bounds and host session persistence using type-safe resolution.
